# vendored-mcp/

Per-MCP vendored sources, built from a pinned commit SHA.

## Pattern

Each subdirectory is one MCP server, with the canonical layout:

```
vendored-mcp/<name>/
  PINNED_COMMIT     # the upstream git SHA we vendored from
  SOURCE_URL        # the upstream repo URL
  TRUST_TIER        # first-party-anthropic | first-party-caia | third-party-vendored
  src/              # vendored source (committed verbatim)
  build.sh          # idempotent reproducible build → ./bin/<name>
  bin/              # built artefact (gitignored, regenerated by CI)
```

## CI verification

`.github/workflows/mcp-vendored-verify.yml` runs on every PR that touches
`vendored-mcp/**`:

1. Reads `PINNED_COMMIT` for each MCP.
2. `git ls-remote <SOURCE_URL>` to fetch the upstream tree at that SHA.
3. `diff -ru` against `vendored-mcp/<name>/src/`.
4. Fails if the local tree drifts from the upstream pinned SHA.
5. Re-runs `build.sh` and asserts the artefact's SHA-256 matches the
   prior CI run (reproducible-build guarantee).

## Why we vendor

Adopted from third-party-paper §C.3 + the v2 update (April 2026 OX
Security disclosure). Vendoring lets us:

- Pin to a SHA we've reviewed.
- Ship without an external `npm install` at orchestrator boot.
- Rebuild deterministically inside the macOS sandbox-exec profile, so
  the binary the agent talks to is the binary we audited.

The allowlist proxy is independent of vendoring — it gates every MCP
spawn regardless of whether the source is vendored. Vendoring is the
defence-in-depth layer we add to a third-party MCP before promoting it
to `trustTier: 'third-party-vendored'`.

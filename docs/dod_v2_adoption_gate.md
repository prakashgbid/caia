> **Date.** 2026-05-16

## Why this addendum

DoD v2 already names three mechanical guardrails:

- **Guardrail #7 — post-merge-deployment-gate** (the merged → deployed enforcer). Partially live: the post-merge queue + watcher landed in PR #452 (INT.1.A1).
- **Guardrail #8 — consumption-probe-daily** (the deployed → consumed enforcer). Still pending.
- **Guardrail #9 — reuse-completeness-reviewer** (the don't-redo-existing-work advisor). Advisory v1 live in PR #453 (INT.1.A4).

The recurring failure mode this addendum addresses sits between #7 and #8: a new utility / package / module / API / external-agent integration is merged into CAIA, but the rest of the codebase that should be calling it isn't updated. The artefact lands, but adoption is left to "someone will get to it" — and the result, measured on 2026-05-16, is ~28-31% of `@chiefaia/*` packages have zero importers outside their own directory, and an additional ~31% have ≤3 real-source importers. The practical "built but not driving value" rate is ~50%.

The fix mirrors how existing guardrails work: make the next step mechanically inescapable. When a PR introduces a new artefact, the substrate identifies every plausible adoption site and either generates the adoption PRs or requires an explicit per-site justification. The originating PR is not "done" until path A (adoption PRs merged) or path B (justifications recorded) holds for every candidate.

## The new rule — Guardrail 8.5: `adoption-enforcement-gate`

Insert into DoD v2 §3 as a new sub-section between §3.b (`consumption-probe-daily`) and §3.c (`reuse-completeness-reviewer`):

### §3.b.2 Guardrail 8.5 — `adoption-enforcement-gate`

**Shape.** A GitHub Actions workflow (`.github/workflows/adoption-enforcement-gate.yml`) plus a launchd cohort (`com.caia.adoption-sweep-hourly`) wrapping the `@chiefaia/adoption-enforcement` substrate. Workflow runs on every PR merge to `develop`; launchd cohort sweeps stragglers hourly.

**The rule.** A PR that introduces or modifies a workspace artefact (a `@chiefaia/*` package, an exported identifier, an MCP integration, or an external-agent manifest) is **not done** until, for every adoption candidate identified by the substrate, one of the following holds:

1. **Path A — automated adoption.** An adoption PR generated by the substrate has been merged into `develop`, replacing the duplicated/legacy custom code at the candidate site with the new artefact.
2. **Path B — operator carve-out.** The candidate site has been explicitly justified as non-adoption in `<repo>/.adoption/justifications/<pkg>.yaml`. Justifications carry `reason`, `expires` (ISO date, default +90 days), and `signed_off_by`. Expired justifications re-open the gate.

A third path does not exist. The point of the gate is to make every adoption decision legible. "Implicit not-adopted" is the failure mode this rule eliminates.

**Coupling to the originating PR.**

- The originating PR is labelled `dod-v2-adoption-pending` immediately on merge by the post-merge runner. The label carries a pointer to `~/.caia/post-merge/work/<sha>/scan.json` and `xref.json`.
- The label is removed by the gate workflow when the adoption ledger (`~/.caia/post-merge/adoption.jsonl`) reports `all_candidates_addressed: true` for that PR's `sha`.
- The label is non-blocking on merge (the originating PR has already merged by the time the gate evaluates). It blocks the originating PR from being reported as "done" in `SESSION_HANDOFF.md` and the DoD-v2 daily audit summary.

**Non-goals.**

- The gate does not auto-merge generated adoption PRs. Default behaviour is `READY` (V1-V6 verification passed) or `DRAFT` (verification failed) for operator review. Auto-merge is a graduation, not a v1 capability.
- The gate does not require zero pending adoption PRs to merge an unrelated downstream PR. The gate scopes per-originating-PR.

## Path B — justification format

`<repo>/.adoption/justifications/<pkg-slug>.yaml`:

```yaml
version: 1
package: '@chiefaia/guardrails-validator'
exemptions:
  - candidate_site: apps/orchestrator/src/safety/tool-result-sanitizer-bridge.ts
    reason: |
      This bridge intentionally implements a narrower sanitization surface
      for tool-result outputs only, not the full agent-input guardrail.
      Adopting @chiefaia/guardrails-validator here would pull in PII /
      secret pattern overhead we don't want on the hot path. Revisit
      when bridge is generalised.
    expires: 2026-08-14
    signed_off_by: prakash
```

The `expires` field is mandatory. The daily hygiene audit reports expired justifications as alerts. This prevents permanent carve-out drift.

## Forward-compat — priority #4

The same gate applies to "external open-source agent integrated everywhere applicable" (priority #4) without re-design:

- A new MCP server registered in `<repo>/.adoption/external-agents.yaml` is an artefact.
- A new external-agent manifest (e.g., aider, autogen, crew) registered there is an artefact.
- The substrate's xref + PR-gen + verify pipeline treats them the same as `@chiefaia/*` exports — only the *detector* in the scan engine differs.

Priority #4's pivot from "build everything in-house" to "integrate external agents everywhere applicable" reduces to: populate `.adoption/external-agents.yaml`, add the detector module, and let the substrate drive adoption. The gate is unchanged.
